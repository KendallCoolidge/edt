Resource bindings

One of the more elegant aspects of EGL is its use of resource bindings, each of which is a set of details for accessing a service or database.

You maintain the bindings in an EGL deployment descriptor, which is external from your logic. That descriptor is used when you are running or deploying your logic and when you are debugging code in the EGL debugger.

Service bindings

If the purpose of a resource binding is service access, the definition is called a service binding, and the main detail is in one of two formats.
  • If the service is deployed on an application server, you can specify a Universal Resource Identifier (URI), as shown in the following example:
    • http://myserver:8080/myproject/restservices/myservice

    Although you can run the deployed service during an EGL debugging session, the EGL debugger does not step into the service.

  • If the service is available in your workspace and was written in EGL, you can use a workspace URI , which is an identifier that points to a workspace location. Here is an example:
    • workspace://mySourceProject/servicepackage.serviceName

    In this case, an internal Test Server enables you to debug the code.

Database bindings

If the purpose of a resource binding is database access, the definition is called a database binding and includes connection detail. The binding might include details or might reference a connection that you defined in Eclipse.

To access a relational database, you include connection details directly in your code, in the DataSource object that is referenced in database-access statements.
